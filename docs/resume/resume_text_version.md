## 滴滴出行 · Agent算法工程师（2025.04至今）

### 项目一：地图DeepSearch Agent

**一句话：** 做了一个能听懂复杂地图问题的AI助手，比如"从公司回家路上帮我找个加油站"，它会自己规划路线、搜索沿途加油站、推荐最优选择。

**核心算法工作：**

1. **用强化学习训练模型的工具调用能力**
   - 问题：开源大模型（Qwen2.5）直接做地图工具调用效果差，经常选错工具或漏填参数
   - 方案：用GRPO强化学习方法训练，设计了5个维度的奖励函数（工具选对没、参数填全没、调用成功没、多步计划合理没、最终回答靠谱没），既有规则打分也有大模型打分
   - 效果：工具调用准确率从72%提到88%，复杂多步任务完成率从65%提到83%
   - 额外验证：对比了DPO方法，GRPO效果好5个点，跟DeepSeek-R1论文结论一致
   - 训练规模：8张A800，LoRA微调，3轮

2. **搭建自动化评测体系**
   - 问题：之前模型好不好全靠人主观判断，没法量化，迭代没方向
   - 方案：参考学术界的AgentBench/ToolBench，搭了一套离线评测：工具选择和参数完整性用规则精确匹配，回答质量用大模型当裁判（Qwen2.5-72B pairwise对比）
   - 效果：大模型裁判跟人工判断一致率87%，接入CI每次更新自动跑评测
   - 价值：模型迭代从"凭感觉"变成"看数据"，上线前有明确的质量门禁

3. **自动化造数据**
   - 问题：地图场景没有现成的Agent训练数据，人工标注太贵
   - 方案：用DeepSeek-V3自动生成各种地图问题（附近搜索、沿途搜索、多点路线等8类场景），配上Web Search搜到的真实POI数据，经过格式检查→执行验证→质量过滤三步筛选
   - 产出：8万多条训练数据，其中3万条SFT数据、5万对偏好数据

4. **参数填充优化**
   - 问题：模型经常漏填关键参数（城市名、坐标），导致工具调用失败要重试
   - 方案：专门造了6类容易出错的样本（参数缺失、城市歧义、品牌词识别等），用课程学习从简单到难训练
   - 效果：参数完整率从78%提到91%，因为不用反复重试了，单次调用耗时从1.8秒降到1.3秒

**工程方面：**

- 搭了ReAct推理框架，简单查询800ms内返回，复杂查询平均3.2步4.5秒内搞定
- 训练后模型推理步骤更精炼，平均少了0.6步，延迟降了12%
- 做了路线规划三阶段链路（召回→语义选点→重算排序），路线采纳率比基线高15%
- Web搜索4路并行，结果可用率从61%提到82%

**业务规模：** 日均50万+次查询

### 项目二：AI出行打车Agent

**一句话：** 在滴滴App里做一个能对话的打车助手，用户说"帮我叫个车去机场"，它理解意图、补全起终点、选路线，直接完成交易。

**核心算法工作：**

1. **意图识别与上下文管理**
   - 设计了渐进式上下文注入：先判断用户要打车还是叫顺风车，再按需加载对应的工具和规则，不把所有工具一股脑塞给模型
   - 效果：Token消耗降35%，意图识别准确率96.2%

2. **Memory系统**
   - 搭了三层记忆：长期画像（用户家/公司地址）、短期状态（当前对话）、按需召回（历史订单）
   - 效果：多轮对话中能正确引用历史信息的比例从41%提到68%

**可观测与体验：**

- 建了全链路监控：首token延迟450ms内，端到端3.2秒内，工具成功率97.5%
- 做了思考流可视化（用户等待时能看到Agent在干什么），流失率降18%

**业务规模：** 日均20万+次对话
